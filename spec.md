2.1 langchain-ai/open_deep_research
2.1.1 项目概览
open_deep_research 是 LangChain 官方推出的深度研究框架，其核心定位是复现并超越
OpenAI Deep Research 的能力。该项目基于 LangGraph 构建，采用层次化的"主管-研究员"（S
upervisor-Researcher）多智能体架构，支持多种搜索工具（Tavily、Serper 等）和
MCP（Model Context Protocol）工具服务器，可配置不同的 LLM
后端。项目强调可定制性和可扩展性，允许用户替换模型、搜索工具、提示词模板等各个组件。
2.1.2 核心架构与流程
该框架的执行流程由 LangGraph 状态图驱动，包含以下关键节点：
1. 澄清交互（clarify_with_user）：系统首先分析用户查询，如果研究范围不够明确，会生
成澄清问题并通过 human-in-the-loop
机制等待用户回答。这是四个实现中唯一内置用户交互环节的方案。
2. 研究简报生成（write_research_brief）：将用户查询（及澄清回答）转化为结构化的研
究简报，明确研究目标、范围和预期产出格式。
3. 研究主管子图（research_supervisor）：这是核心的层次化多智能体模块。主管智能体负
责制定研究策略，通过 think_tool 进行战略反思，然后调用 ConductResearch
委派具体研究任务给多个并行的研究员智能体。
4. 研究员子图（researcher subgraph）：每个研究员智能体是一个独立的子图，包含
researcher 节点（执行搜索）、researcher_tools 节点（工具调用）和 compress_research
节点（压缩研究发现）。研究员可以使用搜索工具、MCP 工具和 think_tool。
5. 研究压缩（compress_research）：每个研究员完成搜索后，将原始发现压缩为简洁的摘
要，以管理上下文窗口限制。这一机制使得系统可以在有限的上下文内处理大量信息。
6. 最终报告生成（final_report_generation）：主管收集所有研究员的压缩摘要后，综合生
成最终的结构化研究报告。
2.1.3 关键特性
• 层次化多智能体：主管-研究员的两层结构，支持并行研究（max_concurrent_research_u
nits 可配置）。
• 战略反思：通过 think_tool 让智能体在行动前进行显式推理和策略调整。
• 渐进式截断：当上下文接近 token
限制时，自动截断较早的研究内容，保证系统不会因上下文溢出而崩溃。
• 重试机制：关键节点使用 with_retry 包装，提高执行稳定性。
• MCP 工具集成：支持通过 MCP 协议接入外部工具服务器，极大扩展了系统能力边界。

2.2 assafelovic/gpt-researcher
2.2.1 项目概览
gpt-researcher 是最早的开源深度研究框架之一，由 Tavily AI 创始人Assaf Elovic
创建。该项目灵感来源于 Princeton 大学 2023 年的论文"Plan-and-Solve Prompting"，采用"规
划-执行"范式，支持标准研究和深度研究两种模式。项目拥有最成熟的工程化实践，包括多种检索
器（Tavily、Serper、DuckDuckGo
等）、多种抓取器、详细的文档和活跃的社区。标准研究模式约 3 分钟、成本约 0.1
美元；深度研究模式约 5 分钟、成本约 0.4 美元（使用 o3-mini high）。
2.2.2 核心架构与流程
标准研究模式（Standard Research）
标准模式采用线性的"规划-执行"流程：
1. 智能体创建：根据查询内容创建领域特定的智能体配置。
2. 规划：规划器（Planner）生成一组研究问题，覆盖查询的各个维度。
3. 并行抓取：对每个研究问题，爬虫智能体（Crawler Agent）搜索并抓取相关网页资源。
4. 摘要与引用：对每个抓取的资源进行摘要，并跟踪信息来源。
5. 过滤与聚合：过滤低质量内容，聚合所有摘要后的信息源。
6. 报告生成：基于聚合的信息生成最终报告，包含内联引用。
深度研究模式（Deep Research）
深度模式引入了树状探索结构，通过 breadth（广度）和
depth（深度）两个参数控制探索规模：
• breadth（默认 4）：每个层级并行探索的研究路径数量。
• depth（默认 2）：递归探索的层级数。
• concurrency（默认 4）：最大并行操作数。
深度模式的核心是一个递归函数，在每一层生成多个查询，并行执行搜索和摘要，然后将发现
作为下一层的上下文继续探索。系统通过 async/await实现并发处理，并具备智能的上下文管理（
跨分支共享发现）和进度跟踪机制。错误处理方面，失败的查询会被跳过而非终止整个流程。
2.2.3 关键特性
• 多检索器支持：集成 Tavily、Serper、DuckDuckGo、Bing 等多种搜索引擎 API。
• 多抓取器支持：支持 BeautifulSoup、Selenium、Playwright 等多种网页抓取方式。
• 成本优化：规划阶段使用 gpt-4o-mini，综合阶段使用 gpt-4o，平衡成本与质量。
• 偏见降低：通过聚合多个信息源来降低单一来源的偏见。
• 内联引用：报告中直接标注信息来源，便于验证。
• 详细文档：拥有最完善的文档和教程，社区活跃度高。

3. 工具智能体（Tool Agents）：并行执行选定的工具，获取新信息。
4. 观察综合智能体（Observations
Agent）：综合新获取的信息，提供对研究策略的思考和建议。
5. 循环：回到步骤 1，继续识别新的知识盲区，直到满足终止条件（最大迭代数、最大时间或知

DeepResearcher（长报告模式）

1. 规划智能体（Planner

2. 并行 IterativeResearcher：为每个章节并行启动一个独立的IterativeResearcher
实例，各自深入研究对应主题。
3. 校对智能体（Proofreader Agent）：将所有章节的研究结果编译为一份连贯的长篇报告，
确保各章节之间的逻辑一致性和内容衔接。

4.3 关键设计决策
在六阶段模型的基础上，深度研究系统的设计者需要在以下关键决策点上做出选择，这些选择
定义了系统的特性和能力边界。
4.3.1 规划策略：静态 vs 动态
静态规划在研究开始时一次性生成所有子任务，执行过程中不修改计划（gpt-researcher
标准模式）。动态规划在执行过程中根据发现不断调整计划（agents-deep-research 的知识盲区
驱动）。动态规划具有更强的自适应能力，但实现复杂度更高，且可能引入规划不稳定的问题。
4.3.2 迭代控制：参数驱动 vs 质量驱动
参数驱动通过固定参数（breadth、depth、max_iterations）控制迭代规模（gpt-research
er、dzhng/deep-research）。质量驱动通过评估当前知识的充分性来决定是否继续迭代（agents
-deep-research 的知识盲区覆盖判断）。质量驱动更智能但需要可靠的质量评估机制。
4.3.3 上下文管理：累积 vs 压缩 vs 分层
累积策略直接将所有发现放入上下文（dzhng/deep-research），简单但易溢出。压缩策略对
发现进行摘要后再放入上下文（open_deep_research的 compress_research），有效管理上下
文但可能丢失细节。分层策略维护不同层级的记忆（工作记忆、情节记忆、语义记忆），在上下文
管理和信息保留之间取得平衡，但实现复杂度最高。
4.3.4 并行策略：串行 vs 并行 vs 混合
串行策略逐个执行搜索和处理（dzhng/deep-research），简单但效率低。并行策略同时执行
多个搜索和处理（gpt-researcher 的 async/await），效率高但需要并发控制。混合策略在不同层
级采用不同策略——如主管串行决策、研究员并行执行（open_deep_research），平衡了效率和
可控性。

第五章 MiroFlow 论文的关键启示
5.1 MiroFlow 概述
MiroFlow（arXiv:2602.22808）是一个高性能的开源智能体框架，专注于深度研究任务。该
框架在 GAIA、BrowseComp、HLE、xBench-DS、FutureX等多个权威基准测试上达到了
SOTA（State-of-the-Art）性能，超越了此前的开源方案和部分商业产品。MiroFlow
的核心技术贡献在于其三层层次化架构和三项关键技术创新，为新框架的设计提供了重要启示。
5.2 三层层次化架构
MiroFlow 采用控制层、智能体层和基础层的三层架构。控制层负责整体工作流的编排和调度
，包括任务分解、智能体分配和结果聚合。智能体层包含执行具体任务的智能体节点，每个节点是
一个可独立调用的智能体。基础层提供底层能力支持，包括 LLM
调用、工具接口和记忆管理。这种分层设计实现了关注点分离，使得各层可以独立演化和优化。
5.3 三项关键技术创新
5.3.1 智能体图（Agent Graph）
MiroFlow 使用有向图结构来组织智能体协作。图中的节点代表智能体，边代表数据流向和控
制依赖。这种图结构比线性的流水线或固定的状态机更加灵活——可以根据任务需求动态调整图的
拓扑结构，支持分支、合并、循环等复杂模式。智能体图使得复杂的工作流可以被清晰地表达和管
理，同时支持并行执行和条件路由。
5.3.2 鲁棒工作流（Robust Workflow）
MiroFlow 的鲁棒工作流包含三项关键机制：消息规范化、重试机制和故障隔离。消息规范化
对智能体之间的通信消息进行标准化处理，减少了 LLM 输出的随机波动对工作流稳定性的影响。
重试机制对失败的智能体调用进行自动重试，提高了执行的可靠性。故障隔离确保单个智能体的失
败不会导致整个工作流崩溃，失败会被隔离在局部范围内处理。这三项机制共同保证了系统在面对
LLM 不确定性、网络故障和异常输入时的稳定性。
5.3.3 重推理模式（Heavy-Reasoning Mode）
对于复杂问题，MiroFlow 采用集成+验证的重推理模式。集成阶段让多个智能体独立解决同一
问题，生成多个候选答案。验证阶段通过专门的验证智能体对候选答案进行交叉检验，选择最一致
、最可靠的答案。这种模式类似于"多模型投票"但更加精细——验证不仅比较最终答案，还比较推理
过程和证据支持。重推理模式显著提高了复杂问题的一致性和可靠性，是 MiroFlow
在基准测试上取得 SOTA 性能的关键因素之一。

第六章 AURA 新框架设计与实现计划
6.1 设计理念与目标
AURA（Adaptive Unified Research Architecture，自适应统一研究架构）的设计目标是构
建一个在自适应能力、结果质量、执行鲁棒性和成本效率四个维度上全面超越现有方案的开源深度
研究框架。AURA 的设计融合了四个开源实现的优秀实践和 MiroFlow
论文的关键技术贡献，同时引入多项创新设计以突破现有方案的局限。
6.1.1 四大设计目标
1. 自适应（Adaptive）：系统应根据查询复杂度和中间发现动态调整研究策略，而非依赖固定
参数。借鉴 agents-deep-research
的知识盲区驱动模式，并进一步引入质量评分机制实现"质量驱动"的迭代控制。
2. 统一（Unified）：提供统一的接口和架构，支持从快速研究到深度研究、从短报告到长报
告的多种场景，无需切换不同框架。借鉴gpt-researcher 的多模式设计和
agents-deep-research 的双模式设计。
3. 鲁棒（Robust）：内置消息规范化、重试机制和故障隔离，保证系统在面对 LLM
不确定性、网络故障和异常输入时的稳定性。借鉴 MiroFlow 的鲁棒工作流设计。
4. 高效（Efficient）：通过模型分级、并行处理和智能缓存优化成本和延迟。借鉴
gpt-researcher 的成本优化策略和agents-deep-research 的三层模型分级。
6.1.2 核心创新点
AURA 相对于现有方案的五大核心创新点：
• 质量驱动的自适应迭代：引入显式的质量评估器，对当前知识的充分性和可靠性进行评分，
基于评分决定是否继续迭代及迭代方向。
• 三层记忆架构：工作记忆（当前迭代上下文）、情节记忆（历史迭代发现）、语义记忆（跨
研究知识库），实现高效的上下文管理。
• 引用图与交叉验证：构建"声明-来源"的引用图，对关键声明进行多源交叉验证，借鉴
MiroFlow 的重推理模式。
• 渐进式报告生成：报告在研究过程中渐进式构建，而非在最后一次性生成，提高效率并支持
中途检查点。
• 成本感知路由：根据任务复杂度自动路由到合适的模型层级，并维护成本预算，在质量与成
本间动态平衡。

AURA 采用三层架构，借鉴 MiroFlow 的分层设计但进行了扩展和优化。

• 工作记忆（Working Memory）：存储当前迭代的上下文信息，包括当前子任务、搜索结

• 情节记忆（Episodic Memory）：存储历史迭代的发现摘要，按迭代顺序组织。情节记忆

• 语义记忆（Semantic Memory）：存储跨研究的结构化知识，包括实体、关系和事实声明
。语义记忆使得系统可以积累和复用知识，避免重复研究相同主题。这是面向长期使用的高级
特性。
6.3.5 基础层：引用图与交叉验证
AURA 引入引用图（Citation Graph）来管理研究发现的可信度。引用图是一个二部图，一侧
是声明（Claim），另一侧是来源（Source），边表示"某来源支持某声明"。当一个声明被多个独
立来源支持时，其可信度较高；当来源之间存在矛盾时，系统会标记冲突并触发 Verifier
智能体进行深入验证。这一机制借鉴了 MiroFlow
的重推理模式，但将其从"多智能体投票"扩展为"多来源交叉验证"，更适合深度研究场景。
6.3.6 基础层：鲁棒性保障
AURA 将 MiroFlow 的鲁棒工作流设计作为基础层能力，包含三项机制：
• 消息规范化：所有智能体间通信采用结构化 JSON Schema，对 LLM
输出进行解析和验证，不符合 Schema 的输出会被自动修复或重试。这减少了 LLM
输出随机性对系统稳定性的影响。
• 重试机制：所有外部调用（LLM API、搜索
API、网页抓取）都包装在重试机制中，支持指数退避和最大重试次数配置。
• 故障隔离：单个智能体或子任务的失败被隔离在局部范围内，不会导致整个工作流崩溃。失
败的子任务会被标记并跳过，Orchestrator可以决定是否重新规划替代路径。

7.4 风险与缓解
AURA 的实现面临以下主要风险，需要提前制定缓解策略：
• 质量评估器的设计难度：多维度质量评分的权重设定和阈值调优需要大量实验。缓解策略：
先实现简单的覆盖度评估，逐步引入其他维度；使用人类标注数据校准评分模型。
• 三层记忆的工程复杂度：三层记忆的交互和同步可能引入复杂的状态管理问题。缓解策略：
先实现工作记忆和情节记忆，语义记忆作为后续扩展；使用成熟的状态管理库。
• 成本控制的平衡：过于激进的成本控制可能影响质量。缓解策略：设置质量下限，成本控制
不得低于质量下限；提供多种预设配置（经济、标准、深度）供用户选择。
• 多模型兼容性：不同 LLM 的能力和接口差异可能影响一致性。缓解策略：使用 LiteLLM
统一接口；为关键智能体提供模型特定的提示词优化。
